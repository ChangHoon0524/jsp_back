get/post방식 -> forward/redirect 방식 -> jsp 태그 -> git-hub 공용 레포지토리

1. get과 post 방식
	폼 태그 안의 method 속성
	
1) get
	주소에 데이터를 queryString 형태로 추가하여 서버로 전송
	?name=값&age=값
	속도가 빠름, 보안성 x, 검색이나 페이지 등에 사용됨

2) post
	데이터가 요청 본문(request body)에 포함되어 서버로 전송
	속도가 느림, 보안성 get보다 강함, 회원가입/로그인/게시글 등에 사용

2. forward와 redirect
1) forward
	request.getRequestDispatcher("파일경로").forward(request, response);
	서버가 받은 요청을 그대로 유지하면서 서버 내부에서 다른 리소스로 이동
	클라이언트의 요청 데이터와 응답 데이터가 유지됨

특징	1. URL : 변화가 없다
	2. 보안 : 요청 데이터가 유지되므로 서버 내부 로직 처리 후 민감한 정보도 안전하게 처리 가능
		클라이언트에게 노출되지 않아 안전한 내부 작업에 적합
	3. 서버 부담 : 서버 내부에서만 작업하므로 부담이 적음

2) redirect
	rresponse.sendRedirect("URL경로");
	서버가 클라이언트에게 새 URL로 이동하라고 요청
	클라이언트는 새로운 요청을 만들어서 해당 URL로 접근

특증	1. URL 변화가 있다 : 브라우저의 URL이 변경되므로 사용자가 이동한 경로를 알 수 있다
	2. 데이터 유지 불가 : 기존 요청 데이터는 소멸 -> 새로운 요청만 서버 전달
	3. 보안 : 민감한 데이터를 처리한 후 클라이언트의 새 요청으로 이동시켜 데이터 재전송 방지
		보안성이 향상되며, 민감한 데이터가 URL에 포함되지 않음
	4. 서버 부담 : 클라이언트가 새 요청을 하므로 서버 부담이 증가한다

HTTP 기본 구조
	클라이언트 -> 요청(request) -> 서버 -> 응답(response) -> 클라이언트
	요청 : 브라우저가 서버에 보내는 것
	응답 : 서버가 브라우저에 보내는 것

request / response
- request
	클라이언트가 보낸 모든 정보가 들어있는 객체
	포함 내용 : 폼데이터, URL, GET/POST 방식, 헤더, 쿠키

- response
	서버가 클라이언트에게 보내는 결과를 담는 객체

getParameter / setAttribute
- getParameter()
	사용자가 보낸 값 읽기
	항상 String 반환

- setAttribute()
	서버가 만든 값을 저장
	Object 저장 가능
	같은 요청 안에서만 유효
	
forward / redirect
- forward 
	요청 1번
	request 유지
	주소창 안 바뀜
	setAttribute 사용 가능
	쿼리스트링 또는 session 사용
	브라우저 -> 서블릿 -> jsp
	(요청 1번, request 유지)

- redirect
	요청 2번
	request 새로 생성
	주소창 변경됨
	setAttribute 사용 불가
	쿼리스트링 또는 session 사용
	브라우저 -> 서블릿 -> 브라우저 -> JSP
	(요청 2번, request 초기화)

+) redirect 시 주의 사항
	URLEncoder.encode(name, "UTF-8")
